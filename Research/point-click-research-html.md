# Executive Summary

> Status (2026-03-03): Working draft only. Contains unresolved citation placeholders and must not be treated as final implementation evidence.

- **Interactive browser inspirations:** We identified dozens of **web-based “game-like” sites** (e.g. scroll/quiz experiences, hidden-click adventures) that use **point‑and‑click hotspots** and animated content. Examples include Gucci’s Off The Grid quiz, the Rick & Morty portal site, Bruno Simon’s 3D portfolio, and others【89†L48-L56】【110†L305-L314】. These demonstrate how to weave narrative and interactivity into a marketing page.
- **Fixed-camera horror games:** We catalogued indie survival-horror titles (Don’t Escape 4, Stasis, Rusty Lake/Cube Escape series, Tormented Souls, The White Chamber, etc.) that use **static backgrounds with interactive overlays** (often isometric or fixed angles). They combine puzzle gating, inventory loops, and atmospheric art【91†L151-L156】【93†L39-L45】. Their design patterns (e.g. limited inventory items, puzzle hotpots, event triggers) can be “stealed” for our MVP (see **Reference Matrix**).
- **Enemy presentation methods:** We compared three techniques for zombie animations: (A) **Sprite-sheet animations**, (B) **Transparent video overlays** (e.g. WebM with alpha channel), and (C) **2.5D layered sprites** (billboarded 2D or simple 3D models over a static scene). Sprites use VRAM and are CPU-light (multiple sprites barely cost extra CPU)【99†L207-L215】, but higher VRAM. Video needs ~3× more CPU and uses ~20 MB memory for a few-second clip【99†L179-L187】, though it can compress longer animations effectively. For mobile, sprite sheets are universally supported; videos may need user gestures or fallback on iOS. We recommend sprite sheets (plus occasional short videos for splashiness).
- **Hitbox sync:** For **sprites**, synchronizing a hitbox (e.g. for shooting) is straightforward: align coordinates with the animated frame index. For **video overlays**, use the `requestVideoFrameCallback` API【39†L?】 (a modern MDN-recommended approach) or time-based events to update hitboxes per frame. Pre-rendered hitboxes (e.g. an image mask per frame) can be used. We’ll preload either all frames (if sprite) or a short video, and fallback to reduced animation on slow devices (see below).
- **Preload strategy:** Use HTML’s `preload` for critical assets (camera background, first zombie animation) and lazy-load others (later rooms, extra zombies). Consider **source-set variants** and `AVIF`/`WebP` for background images (modern browsers)【39†L?】 to minimize size. Video should have `preload="none"` (or a low-priority) unless it’s the first encounter.
- **Performance budgets:** We set tight asset budgets to hit ~30 fps and fast loads. **Per-room background**: ~0.5–1 MB each (preferably <800 KB by using AVIF/WebP). **Initial payload**: aim ≤2 MB total (one room + engine). **JS**: under ~100 KB after compression. **Audio**: limited to short loops/SFX (1–2 MB total). **FPS target**: ~30 fps stable on mid-tier devices. **LCP (Largest Contentful Paint)** target <2 s on 3G (important for assignment SEO). We’ll use responsive images and compression (AVIF/WebP pipelines), and a caching policy via static hosting (immutable cache headers) to meet these budgets.
- **Tooling stack:** We recommend **Vite** (fast build, dev server, static output) as the dev build tool【107†L203-L212】. For hosting and analytics, **Cloudflare Pages + Cloudflare Web Analytics** is ideal: free static hosting with built-in privacy-friendly analytics【105†L382-L390】. For forms, we’ll use a free solution (e.g. Netlify Forms or Formspree) to handle contact submissions. Audio can be handled via **Howler.js** (Bruno Simon’s site uses Howler for ambient sound【85†L235-L238】).
- **Development stack:** A simple “free-stack” might be: Vite + GitHub (for code) + **Cloudflare Pages** (deploy via GitHub Actions) + **Cloudflare Web Analytics** for site metrics【105†L382-L390】 + a static form service (Formspree, etc.) + Howler.js for audio. This avoids server costs. Alternative: Vercel or Netlify (also free tier) with their form/analytics.
- **MVP scope (P0):** Must-have: 5 static pages (landing, market, devlog, etc) tied together with clickable navigation. Implement **hotspot click loops** (gun pickup, zombie fight, loot pickup, market interface). Load **pre-rendered backgrounds** (like RE1 style) with an animated zombie sprite. Include one simple puzzle (e.g. unlocking a door with a found key). Integrate basic analytics (Cloudflare) and a contact form (free tier).
- **Polish (P1):** Add extra features like inventory UI polish, multiple weapons, advanced animations, or Easter eggs. Possibly incorporate limited audio (ambient sounds, gun shot) via Howler. Optimize for various devices (low-quality fallback mode, responsive images).
- **Next steps:** Begin by cataloguing and bookmarking top interactive sites (the Reference Matrix below). Set up the Vite project and a test page with one hot-spot. Iteratively build one room at a time within the performance budget. Use the decision table to choose between graphics methods (sprite vs video) and hosting options.

# Key Decisions (What / Why / Risk / Fallback)

| **Decision**                               | **Why**                                                                                                                         | **Risk**                                                                         | **Fallback**                                                                                      |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| **Use sprite-sheets for zombie**           | Lowest CPU, multi-instance friendly【99†L207-L215】; easy hitbox sync by frame index.                                                | Larger VRAM; many frames may bloat bundle.                                             | If VRAM is too high, fall back to shorter loops or compressed video for non-critical scenes.            |
| **Use transparent video for animation**    | Smaller file size for long animations; smooth 60fps.                                                                                  | High CPU (≈0.8ms/frame) and memory (≈20MB)【99†L179-L187】; limited mobile support. | Limit video to brief clips (gun pickup only) or preload key frames; use image fallback on weak devices. |
| **Use 2.5D layering (static BG + sprite)** | Best compromise: static background (small PNG) + animated sprite layers.                                                              | Hitbox logic per layer is manual.                                                      | If complexity rises, implement video for complex multi-sprite scenes with HTML overlay.                 |
| **Frame/time sync approach**               | Using `requestVideoFrameCallback` (MDN) for video frames【39†L?】; for sprites, use `requestAnimationFrame` tied to frame index. | `requestVideoFrameCallback` has limited browser support (~Chrome/Edge)【39†L?】.    | Fallback to `timeupdate` events or setTimeout loops if needed (less precise).                         |
| **Preload strategy**                       | Preload first room assets (background, initial sprite). Lazy-load others with `<img loading="lazy">` / code split.                  | Large initial load might hurt LCP.                                                     | If initial load too slow, defer animations or reduce image size further.                                |
| **Image formats**                          | Use**AVIF/WebP** (80% smaller than JPG/PNG)【39†L?】; generate responsive sizes.                                               | Some older browsers lack AVIF.                                                         | Provide JPEG fallback or `<picture>` with `type="image/webp"` and `type="image/png"`.             |
| **Static hosting stack**                   | Vite + Cloudflare Pages: zero backend cost, easy deploy, built-in SSL.                                                                | Learning curve on deployment scripts if new.                                           | Alternatively use GitHub Pages (with Actions) or Netlify (includes form handling).                      |
| **Contact form solution**                  | Formspree or Netlify Forms: no server code, quick setup.                                                                              | Limits on free plan submissions; spam risk.                                            | Use captcha or email-only fallback; or accept emails via a generic mailto link if needed.               |
| **Analytics**                              | Cloudflare Web Analytics (free, privacy-first)【105†L382-L390】.                                                                     | Cloudflare rollout may require waiting; or limited features.                           | Temporarily use Google Analytics or Plausible for initial data, then migrate.                           |
| **Audio library**                          | Howler.js (robust, widely used)【85†L235-L238】.                                                                                     | Library size (~19 KB gzipped); might exceed small budget.                             | Use raw HTML5 `<audio>` for simple effects if bundle size is critical.                                |

# Architecture & Tech Stack (Mermaid)

```mermaid
flowchart TD
    DevPC["Developer PC (Vite Dev)"] -->|npm run build| StaticSite
    StaticSite["Static Site (dist/)"] --> CDN["Host: Cloudflare Pages (or Netlify)"]
    CDN --> Browser["Client Browser"]
    Browser -->|GET assets| CDN
    Browser --> CFAnalytics["Cloudflare Web Analytics"]
    Browser --> Forms["Contact Form API (Formspree/Netlify)"]
    Browser --> Audio["Howler.js Audio Library"]
    StaticSite -->|source-maps, manifest| CDN
  
    subgraph Player["Game Engine"]
        BrowserJS["Engine JS (Vite bundle)"]
        Sprites["Sprite Sheets (PNG)"]
        Backgrounds["BG Images (AVIF/WebP)"]
        Sounds["Audio files (MP3/OGG)"]
        VideoOverlay["Zombie overlay video (optional)"]
    end
    CDN --> Player
    Player -->|controls| Browser
  
    note right of CDN: Assets cached globally with immutable headers
    note right of CFAnalytics: Privacy-first, no cookies【105†L382-L390】
    note right of Forms: Free-tier backend, API receives POST submissions
```

**Notes:** The diagram shows our *static site pipeline*: Vite builds assets (JS, images, audio) pushed to a CDN-backed Pages host. The client loads the site (interactives are JS-driven). Cloudflare Analytics collects hits at edge (no extra script)【105†L382-L390】. The contact form hits a third-party API (no server needed). Howler.js runs in browser for sound. We serve AVIF/WebP images (with JPEG fallback) and sprite sheets or video via standard `<img>`/`<video>` tags or canvas. Caching is set to “immutable” on these static resources to maximize reuse.
